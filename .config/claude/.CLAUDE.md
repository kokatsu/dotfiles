# Claude Code Configuration

<important if="you are saving memories or context to files">

## Memory Instructions

When saving any memories or context, always write in English regardless of the conversation language.

</important>

## User Interaction

When responding to queries, follow this priority:

1. **Answer questions first** (especially messages ending with ?)
2. **Clarify ambiguous requests** before making changes
3. **Implement changes only when explicitly requested** (using verbs: implement, create, fix, add, refactor)

<important if="you are deleting or removing files">

## File Deletion

Use `gomi` instead of `rm` for all file deletion operations.

</important>

## Response Quality

Do not use unverified claims to support conclusions. If unsure, say so.
When providing version numbers, CLI command syntax, or technical claims, always verify against actual source code, documentation, or tool output before stating them as fact. If unable to verify, explicitly say so.
When mentioning packages or libraries, always search for the latest information (via web search, official docs, etc.) to verify accuracy before responding.

<important if="you are installing tools, packages, or dependencies">

## Environment

For Nix-managed environments, always use the Nix approach for installing tools (add to flake/config) rather than suggesting direct package manager installs (npm -g, pip, cargo install, etc.).

</important>

<important if="you are writing configuration files, API calls, or using options from documentation">

## Documentation Verification

When writing configuration files or code based on official documentation/references, verify that each option/API is **actually supported in the target context** (ecosystem, runtime, plan tier, etc.). Being documented does not mean being available in every context.

</important>

<important if="you are searching the web or verifying information online">

## Web Search Fallback

When the WebSearch tool fails or returns insufficient results, use the `browser-research` skill (agent-browser CLI) as a fallback to fetch and summarize web content.

</important>

<important if="you are running tests through wrappers (docker run, docker compose run, ssh, or project-specific CLIs)">

## Test Output Compression

The `cc-filter` PreToolUse hook only rewrites commands that begin with a known test runner (`cargo test`, `rspec`, `bundle exec rspec`, `bun test`, `jest`). When these run inside a wrapper, the hook passes through unchanged — manually append ` 2>&1 | cc-filter stream -k <kind>` to compress output.

| Inner command | `<kind>` |
|---|---|
| `cargo test` | `cargo-test` |
| `rspec` / `bundle exec rspec` | `rspec` |
| `bun test` | `bun-test` |
| `jest` | `jest` |

Example: `docker run --rm app cargo test 2>&1 | cc-filter stream -k cargo-test`

</important>

## Design Principles

Always prefer simplicity over pathological correctness. YAGNI, KISS, DRY. No backward-compat shims or fallback paths unless they come free without adding cyclomatic complexity.
