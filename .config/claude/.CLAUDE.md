# Claude Code Configuration

<important if="you are saving memories or context to files">

## Memory Instructions

When saving any memories or context, always write in English regardless of the conversation language.

</important>

## User Interaction

When responding to queries, follow this priority:

1. **Answer questions first** (especially messages ending with ?)
2. **Clarify ambiguous requests** before making changes
3. **Implement changes only when explicitly requested** (using verbs: implement, create, fix, add, refactor)
4. **Decide small, reversible choices autonomously** — for minor decisions (naming, defaults, which of equivalent approaches), pick a reasonable option and note it rather than asking. Reserve clarification for genuine ambiguity, scope changes, and destructive/irreversible actions.

<important if="you are deleting or removing files">

## File Deletion

Use `gomi` instead of `rm` for all file deletion operations.

</important>

## Response Quality

Ground every claim in an authoritative source you checked this session before stating it as fact — not from memory, recall, or training data:

- **Technical facts** (versions, CLI syntax, API options, library behavior) → official docs, source, or tool output
- **File contents, paths, structure** → Read/Glob/Grep
- **Code behavior, call sites, current state** → grep/read the code
- **Historical state, changes, blame** → `git log` / `git blame`
- **External-system state** (CI, issue trackers, dashboards) → query the system

If verification is impractical, say so and label the claim unverified — never present it as fact. Verifying first is cheaper than acting on a wrong claim, so when in doubt, verify.

<important if="you are inspecting git diffs (git diff / show / log -p)">

## Git Diffs

The user's global git config sets `diff.external = difft` (difftastic), which replaces every git diff with a structural, side-by-side rendering that is hard to read when captured as tool output. Always pass `--no-ext-diff` to get a standard unified diff: `git diff --no-ext-diff`, `git show --no-ext-diff`, `git log -p --no-ext-diff`. This is the only reliable way to bypass the external diff (an empty `diff.external=` override errors out). The user's interactive difftastic experience is left untouched.

</important>

<important if="you are installing tools, packages, or dependencies">

## Environment

For Nix-managed environments, always use the Nix approach for installing tools (add to flake/config) rather than suggesting direct package manager installs (npm -g, pip, cargo install, etc.).

</important>

<important if="you are writing configuration files, API calls, or using options from documentation">

## Documentation Verification

When writing configuration files or code based on official documentation/references, verify that each option/API is **actually supported in the target context** (ecosystem, runtime, plan tier, etc.). Being documented does not mean being available in every context.

</important>

<important if="you are searching the web or verifying information online">

## Web Search Fallback

When the WebSearch tool fails or returns insufficient results, use the `browser-research` skill (agent-browser CLI) as a fallback to fetch and summarize web content.

</important>

## Design Principles

Always prefer simplicity over pathological correctness. YAGNI, KISS, DRY. Avoid backward-compat shims or fallback paths unless they add no branching.

<important if="you are editing existing code">

## Surgical Changes

Touch only what the request requires. Clean up only the mess your own changes create.

- Don't "improve" adjacent code, comments, or formatting that the request didn't touch.
- Don't refactor things that aren't broken.
- Match the existing style, even if you'd write it differently.
- Remove imports/variables/functions that *your* changes made unused. Don't delete pre-existing dead code — mention it instead.

The test: every changed line should trace directly to the user's request.

</important>

<important if="you are fixing a bug">

## Reproduce Before Fixing

Before changing code to fix a bug, write a test that reproduces it and fails. Make that test pass, and keep it. A fix without a reproducing test is unverified.

</important>
