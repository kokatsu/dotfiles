# 日本語 Markdown 文章規範

Markdown に日本語の地の文を書くときの規範。コードブロックや設定値、英語で書くファイルには適用しない。
出典: k16shikano「日本語技術文書の文章規範」(gist `fd287c3133457c4fd8f5601d34aa817d`) を一般ドキュメント向けに抜粋。

## 整形

- 複数行のコード、差分、ログ、設定断片はコードブロックに入れる。短い識別子(設定キー、コマンド、ファイル名など)はインラインコードで示す
- 定義や分類の列挙は箇条書きにし、定義語は太字にする。初出は太字、以後の言及は「」で示す
- 用語と説明を並べる箇条書きは `**用語**: 説明` の形で書く
- ダッシュ(em/en/2倍ダッシュ)を地の文や見出しで使わない
  - 同格や補足の挿入 → 丸括弧( )に入れる
  - 言い換えや敷衍 → 句点で二文に分けるか読点でつなぐ
- 中黒(・)を日本語の並列に使わない(固有名詞の内部は可)
- 見出しは単一の自然な句にする。「種別──主題」のような二要素の詰め込みを避ける
- 見出しは扱う対象か答える問いを示す。手順だけの見出し(「○○に戻す」)にしない

## 用語と視点

- 例示は受動態の羅列でなく、行為者を主語にした動作の連なりで書く
- 読者を「あなた」と呼ばず、役割名(「開発者」「利用者」)で書く
- 「AI」「ツール」のような広い語でぼかさず、対象を具体名で書く
- 術語を導入したら以後はその語で通す。曖昧語に後退しない
- 訳語は分野で慣用されている語を選ぶ。人名は原綴りで書く(例: Lehman)
- 読者の行動や検証に使わない固有名は本文に出さない。検索や参照に使うものは残す

## 論証の正確さ

- 因果を主張するときは、その機構(なぜそうなるか)を一文添える
- 「必ず」できるかのように書かず、条件付きで正確に述べる
- 推量や仮定、限定は根拠なく弱めている場合のみ削る。不確実な事実はそのまま残す
- 確認していないことを、確認したかのように滑らかに書かない
- 「AではなくB」と否定するときは、否定の根拠を一文添える

## 冗長の排除

- 同じ主張を言い換えて繰り返さない。一主張は一度だけ書く
- 論理的役割が同じ並列事実は一文にまとめる
- 読者が自力で補える中間段階の説明は書かない
- 接続や評価のためだけの文(「それ自体はよいことである」など)を置かない
- 問答形式の修辞(問いを立てて一語で答える)を使わない
- まだ導入していない概念を先回りして持ち出さない
- 根拠で確定していることは、弱い述語でぼかさず言い切る

## LLM っぽい表現の禁止

「論点を増やさずに、ちゃんと書いている感だけを足す」表現は使わない。

| カテゴリ | 禁止例 |
|---|---|
| 予告と総括 | 「重要なのは〜」「本章では〜を探求する」「まとめると」「〜に他ならない」 |
| 正面から系 | 「正面から扱う」「正面から回収する」「正面から見る」 |
| 空虚な形容 | 「不可欠」「核心的」「鍵となる」「多角的」「包括的」「総合的」 |
| 空虚な動詞 | 「掘り下げる」「深掘りする」「言語化する」「触れる」「言及する」 |
| 接続の型 | 「〜において」「〜という側面から」「さらに」「また」「加えて」の連打 |
| 弱い緩和や称賛 | 「〜と言えるだろう」(根拠なし)、「非常に」「極めて」「大いに」(中身なし) |
